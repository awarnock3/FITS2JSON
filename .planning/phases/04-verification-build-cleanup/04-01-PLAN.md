<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI arguments → CFITSIO open/select | Untrusted file paths and selectors enter the parser boundary |
| Make variables / environment → compiler and linker flags | Build-time inputs can change discovery and link behavior |
| Generated test artifacts → cleanup rules | Verification-generated files cross into deletion logic |
| In-memory JSON → stdout pipe | Downstream consumers can close or fail mid-emission |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | T | `src/fits2json.c` argument/open/select failure path | mitigate | Validate `argc`, keep model-first/close-before-emit flow, route all failures through centralized stderr helpers, and preserve CFITSIO detail output for parser-backed failures. |
| T-04-02 | D | `src/Makefile` dependency discovery | mitigate | Prefer `pkg-config cfitsio`, allow explicit `CFITSIO_CFLAGS`/`CFITSIO_LIBS` overrides, and fail before compilation/linking with a clear prerequisite message instead of confusing toolchain errors. |
| T-04-03 | T | `src/Makefile` clean target | mitigate | Use an explicit generated-artifact list and post-clean assertions so only built binaries, helper outputs, generated fixtures, and caches are removed; never glob checked-in `test/` or `testdata/` sources. |
| T-04-04 | D | stdout write / broken-pipe failure after emission begins | accept | OS pipe semantics can allow downstream consumers to receive bytes before failure is detected, so Phase 4 scopes D-05’s empty-stdout guarantee to usage and pre-emission failures; still convert detectable write failures into exit `1` plus `fits2json:` stderr diagnostics. |
</threat_model>
